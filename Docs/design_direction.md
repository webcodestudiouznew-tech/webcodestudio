# Design Direction

## Назначение документа
Этот файл фиксирует дизайн-код проекта WebCode на основе собранной HERO секции.

Это не abstract moodboard и не список пожеланий. Это рабочая система, по которой должны собираться все следующие секции лендинга:
- визуальный характер;
- сетка и контейнеры;
- типографика;
- цветовая схема;
- стиль компонентов;
- правила анимации;
- правила для desktop и mobile.

Если новая секция визуально спорит с этим документом, ориентиром считается этот документ и уже утверждённая HERO секция.

## Общий образ
WebCode должен выглядеть как современный, уверенный, прикладной digital-продукт для локального бизнеса.

Нужное ощущение:
- надёжный;
- собранный;
- технологичный;
- не холодный;
- коммерчески убедительный;
- визуально дорогой, но без fake premium-подачи.

Необходимый результат:
- пользователь быстро понимает оффер;
- визуал создаёт ощущение системы, а не хаотичной студии;
- интерфейс выглядит как единый продукт, а не как набор независимых блоков.

## Базовый визуальный язык
Основа проекта это тёмная тёплая палитра с золотыми акцентами.

Ключевые характеристики:
- глубокий тёмный фон;
- мягкие золотые glow-облака;
- светлая типографика;
- акцентные золотые кнопки и цифры;
- device-centric композиции;
- минимум лишних контуров и декоративных рамок;
- фокус на сильной иерархии и ритме.

Это не should look like:
- типичный SaaS с фиолетовыми градиентами;
- light minimalist corporate site;
- noisy cyberpunk;
- glossy fake luxury without structure.

## Цветовая система
Проект строится вокруг тёмно-золотой схемы.

### Основной фон
- базовый тёмный тон: близкий к `#252421`, `#2d2b28`, `#23221f`
- фоновая секция не плоская; используется сочетание:
  - linear gradient по вертикали;
  - radial glow в правой части;
  - дополнительные размазанные золотые пятна глубины.

### Акцент
- главный акцент: тёплое золото
- ориентиры по тону:
  - `#efcb65`
  - `#d7b24c`
  - `#d4af4a`
  - `#f1cc6b`

Акцент используется для:
- primary CTA;
- orbit badge;
- числовых значений;
- иконок feature-блоков;
- hover brightness;
- звёзд и glow.

### Текст
- основной текст: white / near-white
- вторичный текст: белый с opacity примерно `56% - 74%`
- tertiary/supporting text: белый с opacity около `56%`

### Границы и поверхности
- бордеры используются очень аккуратно;
- если бордер есть, он тёплый и полупрозрачный, а не холодный серый;
- на mobile допустимы soft surface-cards для упрощения восприятия, но на desktop секции не должны быть завалены карточками без причины.

## Фон и атмосфера
Фон должен работать как среда, а не как однотонная подложка.

Обязательные принципы:
- фон full-width;
- контент поверх него читабелен без дополнительных грязных оверлеев;
- glow должен поддерживать композицию, а не мешать тексту;
- градиенты и свечения концентрируются вокруг ключевых visual-зон.

Что допустимо:
- radial glow;
- лёгкая глубина;
- device mockups;
- редкие декоративные звёзды;
- мягкие тени.

Что не использовать без причины:
- шумные texture overlays;
- тяжёлые сетки на всю секцию;
- большое количество иконографики;
- случайные abstract blobs.

## Шрифты и типографика
Шрифтовая пара проекта:
- `Manrope` для заголовков и ключевых UI-элементов;
- `Inter` для текста, интерфейсных подписей и supporting copy.

### Правила
- заголовки всегда на `Manrope`;
- текстовые блоки и навигация в основном на `Inter`;
- letter-spacing у крупных заголовков отрицательный;
- заголовки выглядят плотными, но не тяжелыми;
- body copy читается легко и имеет воздух.

### Типографическая иерархия
На основе HERO:

#### Header
- бренд `WebCode studio`: `18px` mobile, `20px` desktop
- подпись под брендом: `10px` mobile, `11px` tablet, `12px` desktop
- пункты nav: `14px`
- CTA в header: `14px`

#### Hero
- badge: `13px` mobile, `12px` desktop visual size with compact line-height
- hero h1:
  - `46px` mobile
  - `48px` tablet
  - `60px` desktop
  - `64px` large desktop
- description:
  - `16px` mobile
  - `18px` desktop
- hero CTA:
  - `15px`

#### Feature blocks
- feature title: `16px`
- feature description: `13px - 14px`

#### Stats and badges
- stats value: `20px` mobile, `22px - 24px` desktop
- orbit text: `16px`

### Тон типографики
Типографика не должна быть:
- слишком жирной;
- слишком лёгкой;
- слишком размазанной по вертикали.

Плотность должна ощущаться как intentional editorial/product style.

## Сетка и контейнеры
### Общий принцип
- секция full-width;
- контент секции центрируется внутри общего контейнера;
- на desktop основной ориентир по ширине: `1280px`.
- новые секции должны использовать ту же desktop-ширину контейнера, что и HERO, без дополнительного внутреннего horizontal padding сверх общего container width.

### Hero
- HERO занимает всю ширину экрана;
- на desktop используется общий content width `max-w-[1280px]`;
- header и main content выровнены по одной контейнерной сетке;
- секция на desktop сидит в высоте экрана;
- на mobile работает через `min-h-screen`, а не через жёсткое обрезание.

### Mobile safe paddings
- mobile секции используют безопасные боковые отступы уровня `16px - 20px`;
- tablet/mobile-large могут использовать `24px`;
- внутри fullscreen mobile menu применяется тот же safe padding, а не отдельная хаотичная геометрия.

## Правила композиции
### Desktop
Desktop-композиция должна строиться на двух сильных осях:
- слева: оффер, смысл, CTA;
- справа: visual и образ продукта.

Hero desktop:
- header сверху;
- main area из двух колонок;
- снизу supporting row с feature-блоками, orbit badge и stats.

### Mobile
На mobile секция перестраивается в последовательный vertical flow:
- header;
- badge;
- h1;
- visual;
- description;
- CTA;
- feature blocks;
- stats.

Главный принцип mobile:
- порядок должен быть логичным;
- каждый блок должен быть самостоятельным и легко считываться;
- не переносить desktop-композицию в сжатом виде один в один.

## Стиль компонентов
### Header
Header должен быть лёгким и чистым.

Правила:
- бренд слева;
- nav по центру/справа на desktop;
- locale switcher рядом с header CTA;
- отдельный CTA в header справа;
- mobile menu button без лишней рамки и с чистой геометрией;
- fullscreen mobile menu без dropdown-card эффекта.

### Buttons
Кнопки проекта:
- primary: золотой vertical gradient;
- secondary: тёмная поверхность с тонким тёплым бордером;
- оба варианта должны иметь мягкую тень и аккуратный hover.

Primary CTA:
- всегда выглядит как главное действие;
- используется для `Оставить заявку`.

Secondary CTA:
- выглядит спокойнее;
- используется для `Консультация в Telegram` и похожих вторичных действий.

### Feature cards / feature items
На desktop:
- feature items могут быть без явной карточной оболочки;
- больше похожи на structured info blocks.

На mobile:
- допустим мягкий surface-card стиль с тонкой границей и blur;
- это помогает разбить длинную вертикальную ленту на удобные куски.

### Orbit badge
Orbit badge это фирменный supporting element проекта.

Правила:
- круговой текст + центральный круг;
- центр badge должен использовать бренд-иконку, а не случайный знак;
- badge не должен быть главным героем секции;
- он добавляет ощущение продукта и собранности.

### Stats block
Stats block должен выглядеть как social proof, а не как dashboard widget.

Правила:
- короткое число в акцентной плашке;
- стек аватаров;
- короткая подпись;
- выравнивание и расстояния должны быть спокойными.

## Иконография
Иконки:
- линейные;
- чистые;
- без тяжёлых фонов;
- без случайных контейнеров;
- по цвету подчинены золотому акценту.

Иконка должна усиливать смысл, а не становиться декоративным шумом.

## Motion
Проект должен ощущаться живым, но не перегруженным анимацией.

### Обязательные принципы
- анимация мягкая;
- easing современный, не резкий;
- duration умеренная;
- entrance-анимации должны усиливать порядок, а не отвлекать.

### Что уже принято в HERO
- staggered reveal элементов;
- мягкий reveal/fade background glow;
- scale/fade для visual;
- отдельный hover motion для кнопок и nav;
- пульсирующий glow у декоративных звёзд.

### Правила для следующих секций
- не анимировать всё подряд;
- у каждой секции должен быть 1 главный motion-pattern;
- hover должен быть subtle:
  - лёгкий подъём;
  - лёгкое увеличение яркости;
  - без aggressive scale jumps.

### Accessibility
- поддерживать `prefers-reduced-motion`;
- при отключённой анимации интерфейс должен оставаться полностью читаемым и цельным.

## Desktop / Mobile rules
### Desktop
- больше воздуха;
- меньше карточного дробления;
- сильная двухколоночная композиция;
- nav полноценный и всегда видимый;
- large visual допустим.

### Mobile
- больше структурных карточек только там, где это улучшает чтение;
- кнопки full-width;
- тексты центрируются только там, где это улучшает первый экран;
- меню всегда fullscreen overlay;
- spacing на mobile должен быть визуально, а не только математически достаточным.

Важный принцип:
если блок на mobile выглядит слишком плотным, сначала проверять не только `margin`, но и масштаб соседнего visual-элемента.

## Контентно-визуальные принципы
Каждая секция должна отвечать на один бизнес-вопрос.

Примеры:
- Hero: что это и зачем мне это;
- Для кого: подходит ли это моему бизнесу;
- Как это работает: насколько это просто и прозрачно;
- Тарифы: сколько это стоит и что я получаю;
- Кейсы: есть ли подтверждение качества;
- FAQ: снимаются ли основные возражения;
- Контакты / CTA: что делать дальше.

Внутри секции нельзя смешивать слишком много смыслов.

## Что делать в следующих секциях
Следующие секции нужно собирать так:
- сохранять тот же фон и уровень контраста;
- использовать те же контейнеры и safe paddings;
- опираться на те же акцентные золотые цвета;
- сохранять шрифтовую иерархию;
- не ломать established hover и motion language;
- повторно использовать схожие patterns:
  - section heading;
  - CTA pair;
  - glass / soft-surface cards;
  - metric / proof blocks;
  - linear icon style.

## Что нельзя делать
- уходить в светлый random style в середине лендинга;
- смешивать холодные blue/purple accents с текущей палитрой;
- делать следующие секции слишком “шаблонно SaaS”;
- вставлять случайные карточки и бордеры без функции;
- использовать другой motion-language;
- резко менять радиусы, тени, плотность текста и стиль кнопок;
- собирать mobile layout как уменьшенный desktop.

## Source of truth
При разработке следующих секций source of truth:
1. Этот документ.
2. Уже собранная HERO секция.
3. Текущие design tokens, spacing patterns и interaction patterns из кода.

Если появляется спорное решение, нужно проверять:
- соответствует ли оно тёмно-золотому продуктному характеру;
- поддерживает ли ту же иерархию;
- не выглядит ли оно как другой сайт;
- будет ли оно так же хорошо работать на mobile.
