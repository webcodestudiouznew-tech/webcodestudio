# landing_ui_standards.md

Стандарт для визуальной реализации лендингов WebCode после того, как структура секций уже определена.

`guide_landing_layouts_PRO.md` является главным документом для выбора и логики секций.
Этот документ не дублирует структуру лендинга и отвечает только за visual execution.

## 1. Общий принцип
- Лендинг не должен выглядеть как универсальный шаблонный `SaaS`-сайт без характера.
- Визуальные решения должны быть намеренными: typography, spacing, color accents, card language и CTA hierarchy должны работать как единая система.
- Нельзя добавлять визуальные паттерны только потому, что они "обычно так делаются", если они не усиливают уже выбранную структуру страницы.

## 2. Visual Direction
- Все новые секции должны продолжать уже утверждённый visual language проекта.
- Нельзя вводить новый стиль карточек, теней, радиусов, заголовков или CTA без необходимости.
- Если секция требует акцента, усиливать существующую систему, а не создавать новый визуальный мир внутри страницы.
- Background, gradients, blurs, highlights и decorative details должны поддерживать композицию, а не создавать шум.

## 3. Typography
- Для `h1`, `h2`, `h3`, body и supporting text должна соблюдаться единая типографическая система.
- Заголовки не должны держаться на случайных переносах, случайном `max-width` или агрессивном отрицательном `tracking`.
- Крупные display-заголовки допустимы только там, где это действительно оправдано ролью секции.
- Секционные заголовки должны быть читаемыми, плотными, но не тяжёлыми.
- Карточечные заголовки должны быть визуально слабее `h2`, но сильнее body text.
- Нужно контролировать line-length: слишком широкие текстовые строки ухудшают читаемость, слишком узкие ломают ритм.
- Если локаль требует, допустима адаптация размера, `line-height`, `tracking` и ширины текстового блока.

## 4. Spacing System
- Использовать единую шкалу отступов между секциями, внутри секций, между заголовком и описанием, внутри карточек и CTA-блоков.
- Новые компоненты не должны вводить случайные значения padding и gap без причины.
- Вертикальный ритм секций должен быть согласован по всей странице.
- Плотность карточек, списков и CTA должна быть предсказуемой и сопоставимой между соседними блоками.
- Если в одной секции есть два столбца или две карточки рядом, нужно проверять баланс внутреннего пространства и высот.

## 5. Layout Composition
- Layout каждой секции должен быть устойчивым к изменению длины текста.
- Нельзя проектировать композицию, которая работает только на одном языке или только на desktop.
- Grid и card layouts должны выдерживать реальную контентную нагрузку, а не только короткие демо-строки.
- Если контент в карточке требует больше места, сначала пересматривается сама композиция и распределение пространства, а не только уменьшается шрифт.

## 6. Cards And Surfaces
- Карточки должны иметь согласованные радиусы, внутренние отступы, border treatment и shadow logic.
- Нельзя смешивать в одной плоскости несколько несовместимых card styles.
- Wide, tall и compact cards должны ощущаться частью одной системы.
- Любая карточка должна быть читаема без визуальной перегрузки декоративными элементами.

## 7. CTA Presentation
- На каждом экране должен быть понятен primary CTA.
- Secondary CTA и messenger CTA не должны спорить с основным действием.
- Любой текст рядом с CTA должен либо усиливать ясность, либо снижать тревогу пользователя.

## 8. Content Density
- Текст в `Hero` должен быть коротким, сильным и быстро считываемым.
- Тексты в карточках не должны превращаться в абзацы ради заполнения пространства.
- Если секция выглядит пустой, сначала пересмотреть композицию, а не механически наращивать copy.
- Если текст перегружает layout, сначала адаптировать формулировку, а не ломать визуальную систему.

## 9. Motion
- Анимация должна усиливать hierarchy, reveal и perceived quality.
- Нельзя использовать motion только как декоративный шум.
- Длительность, easing и интенсивность анимации должны соответствовать премиальному коммерческому интерфейсу, а не демонстрационному шаблону.

## 10. Visual QA Before Done
- Проверить, не выглядит ли новая секция слабее соседних по качеству.
- Проверить, нет ли случайных размеров шрифтов, padding или gap.
- Проверить, не держится ли секция на одном "удачном" переносе заголовка.
- Проверить, не спорят ли между собой CTA, карточки, фоны и акценты.
- Проверить, ощущается ли страница как единый продукт, а не набор независимых блоков.
