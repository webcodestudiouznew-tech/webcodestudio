# Гайд по секциям и layout-паттернам современного лендинга

Практическая библиотека секций для проектирования landing page, продуктовых страниц и коммерческих одностраничных сайтов.  
Документ предназначен как рабочий справочник для UX/UI-дизайна, маркетинговой упаковки и сборки структуры сайта.

---

# 1. Как пользоваться этим гайдом

Этот документ нужен не для того, чтобы вставлять все возможные секции в один лендинг.  
Хороший лендинг обычно собирается из **7–12 секций**, где каждая выполняет конкретную задачу в воронке:

**Attention → Problem → Solution → Proof → Offer → Objections → CTA**

То есть:

1. Захватить внимание  
2. Обозначить проблему  
3. Показать решение  
4. Подтвердить ценность  
5. Усилить доверие  
6. Снять возражения  
7. Подвести к действию

# 3. Библиотека секций лендинга

## 3.1 Global / Shell sections

### Announcement Bar
Короткая верхняя полоса с промо, акцией, дедлайном, важным сообщением или анонсом.

**Задача:** быстро донести одно главное сообщение.  
**Когда использовать:** если есть акция, запуск, дедлайн, релиз, подарок, ограничение по времени.

**Типы:**
- Promo Bar
- Urgency Bar
- Release Bar
- Event Bar
- Free Shipping Bar

---

### Header / Top Navigation
Верхняя часть сайта с логотипом, навигацией и CTA.

**Задача:** дать ориентир по сайту и доступ к действию.  
**Когда использовать:** почти всегда.

**Типы:**
- Simple Header
- Sticky Header
- Transparent Header
- Multi-link Header
- Mega Menu Header

---

### Sticky CTA Bar
Фиксированная CTA-панель, которая остается видимой при скролле.

**Задача:** удерживать основное действие всегда под рукой.  
**Когда использовать:** на длинных лендингах, mobile-first страницах, лид-формах.

**Типы:**
- Sticky Top CTA
- Sticky Bottom CTA
- Mobile Sticky CTA

---

### Breadcrumbs
Цепочка навигации.

**Задача:** упростить навигацию.  
**Когда использовать:** редко для лендинга, чаще для SEO-страниц и вложенных продуктовых страниц.

---

### Progress Indicator
Индикатор прогресса прокрутки страницы.

**Задача:** показать пользователю, сколько контента осталось.  
**Когда использовать:** на long-form landing page.

---

## 3.2 Above-the-Fold sections

### Hero Section
Первый экран лендинга, который формирует первое впечатление и объясняет оффер.

**Задача:** за 3–5 секунд донести суть предложения.  
**Когда использовать:** всегда.

**Типы:**
- Centered Hero
- Split-Screen Hero
- Left Copy / Right Visual Hero
- Full-Bleed Hero
- Minimal Hero
- Product Screenshot Hero
- Video Hero
- Interactive Hero
- Form Hero

**Обычно содержит:**
- Headline
- Subheadline
- Primary CTA
- Secondary CTA
- Hero Visual

---

### Value Proposition Block
Блок с основным ценностным предложением.

**Задача:** ответить на вопрос «почему это важно именно мне?».  
**Когда использовать:** если оффер не считывается мгновенно на первом экране.

**Типы:**
- Single Proposition
- Three-Point Value Prop
- Benefit-Led Value Prop
- Outcome-Led Value Prop

---

### Hero Media Block
Главный медиа-элемент первого экрана.

**Задача:** визуально показать продукт, интерфейс, результат или трансформацию.  
**Когда использовать:** если визуал продает лучше текста.

**Типы:**
- Product Mockup
- Dashboard Preview
- Explainer Video
- Device Mockup
- Before/After Visual
- Animation Preview

---

### Trust Strip / Credibility Strip
Узкая секция доверия сразу после Hero.

**Задача:** быстро снизить недоверие.  
**Когда использовать:** почти всегда.

**Типы:**
- Logo Strip
- Rating Strip
- Media Mentions Strip
- Compliance Strip
- “Trusted by” Strip

---

## 3.3 Problem / Pain sections

### Problem Statement Section
Блок, описывающий проблему пользователя.

**Задача:** показать, что продукт решает реальную боль.  
**Когда использовать:** когда аудиторию нужно вовлечь через узнавание проблемы.

**Типы:**
- Single Core Problem
- Multi-Pain Grid
- Symptom List
- Pain Bullets
- Pain Cards

---

### Agitation Section
Усиление последствий бездействия.

**Задача:** показать, почему откладывать решение невыгодно.  
**Когда использовать:** если проблема не ощущается как срочная.

**Типы:**
- Cost of Inaction
- Risk Highlight
- Friction Narrative
- Emotional Agitation Block

---

### Before / After Section
Показ контраста между текущим состоянием и результатом после решения.

**Задача:** визуализировать трансформацию.  
**Когда использовать:** когда продается изменение состояния.

**Типы:**
- Before / After Split
- Transformation Cards
- State Change Visual
- Timeline Transformation

---

### Myth vs Reality
Секция разрушения ложных убеждений.

**Задача:** переобучить пользователя и снять неверные представления.  
**Когда использовать:** в сложных, конкурентных и перегретых нишах.

**Типы:**
- Myth / Truth Grid
- Objection Reframe
- Industry Misconceptions

---

## 3.4 Solution / Offer framing sections

### Solution Overview Section
Краткое объяснение того, как работает решение.

**Задача:** показать саму логику решения.  
**Когда использовать:** если продукт новый, сложный или неочевидный.

**Типы:**
- One-Sentence Solution
- Three-Step Solution
- Modular Solution Overview
- Platform Overview

---

### USP Section
Блок с уникальным торговым предложением.

**Задача:** показать отличие от альтернатив.  
**Когда использовать:** если рынок конкурентный.

**Типы:**
- USP Statement
- Why We’re Different
- Differentiator Grid
- Category Positioning Block

---

### Offer Section
Описание конкретного предложения.

**Задача:** объяснить, что именно получает клиент.  
**Когда использовать:** всегда, когда у страницы есть коммерческий оффер.

**Типы:**
- Service Offer
- Product Offer
- Bundle Offer
- Package Offer
- Subscription Offer
- Lead Magnet Offer

---

### Offer Stack
Слой за слоем показывает состав предложения.

**Задача:** усилить ценность оффера.  
**Когда использовать:** для high-ticket, экспертных и инфопродуктов, а также услуг с большим набором deliverables.

**Типы:**
- Stack List
- Value Stack Cards
- What’s Included Section
- Deliverables Breakdown

---

## 3.5 Features / Benefits sections

### Features Section
Секция функций, возможностей и модулей.

**Задача:** показать, что умеет продукт.  
**Когда использовать:** для SaaS, сервисов, приложений, платформ.

**Типы:**
- Feature Grid
- Alternating Features
- Feature Rows
- Tabs Features
- Accordion Features
- Icon Features
- Screenshot + Copy Features
- Bento Features

---

### Benefits Section
Секция выгод и результатов.

**Задача:** показать, что пользователь получает в итоге.  
**Когда использовать:** почти всегда.

**Типы:**
- Benefit Cards
- Outcome Grid
- Benefit Bullets
- Emotional + Functional Benefits
- Benefit Comparison

---

### Feature-to-Benefit Section
Связка «функция → выгода».

**Задача:** сделать ценность функций понятной.  
**Когда использовать:** если features сами по себе неочевидны пользователю.

**Типы:**
- Two-Column Mapping
- Feature / Benefit Table
- Capability-to-Outcome Grid

---

### Capability Matrix
Матрица возможностей продукта.

**Задача:** структурно показать функциональный охват.  
**Когда использовать:** для сложных B2B-продуктов и enterprise-решений.

**Типы:**
- Matrix Table
- Category-Based Capability Grid
- Technical Capability Block

---

## 3.6 How It Works / Process sections

### How It Works Section
Пошаговое объяснение механики.

**Задача:** ответить на вопрос «как это работает?».  
**Когда использовать:** почти всегда, если есть действие, процесс или onboarding.

**Типы:**
- 3-Step Process
- 4-Step Process
- Timeline Process
- Circular Process
- Numbered Steps
- Visual Workflow

---

### Process Section
Процесс выполнения услуги или внедрения решения.

**Задача:** показать этапность и порядок работы.  
**Когда использовать:** для агентств, студий, сервисного бизнеса, консалтинга.

**Типы:**
- Discovery → Design → Delivery
- Onboarding Process
- Implementation Process
- Production Pipeline
- Fulfillment Process

---

### Workflow Section
Расширенная схема пользовательского или операционного процесса.

**Задача:** показать end-to-end сценарий.  
**Когда использовать:** в SaaS, automation, AI-инструментах.

**Типы:**
- User Workflow
- Team Workflow
- Automation Workflow
- End-to-End Workflow

---

### Onboarding Section
Описание начала работы.

**Задача:** снизить страх старта.  
**Когда использовать:** если продукт кажется сложным.

**Типы:**
- Quick Start
- Setup Flow
- First Week Experience
- Zero-to-Value Flow

---

## 3.7 Use Cases / Audience Fit sections

### Use Cases Section
Сценарии использования продукта.

**Задача:** показать, где и как продукт применим.  
**Когда использовать:** если у продукта несколько сценариев или сегментов.

**Типы:**
- Use Case Cards
- Industry Use Cases
- Role-Based Use Cases
- Job-to-be-Done Use Cases
- Scenario Grid

---

### Who It’s For Section
Для кого подходит решение.

**Задача:** дать пользователю быстро узнать себя.  
**Когда использовать:** когда есть несколько типов аудитории.

**Типы:**
- Audience Segmentation
- Persona Cards
- Business Type Grid
- Role Fit Section

---

### Who It’s Not For Section
Для кого решение не подходит.

**Задача:** отсеять нерелевантную аудиторию и усилить качество лидов.  
**Когда использовать:** в премиум, B2B, high-ticket нишах.

**Типы:**
- Fit / No-Fit Matrix
- Qualification Block
- Disqualifier Section

---

### JTBD Section
Формулировка через Jobs To Be Done.

**Задача:** описать продукт через реальные задачи пользователя.  
**Когда использовать:** если продается не функция, а выполнение работы/задачи.

**Типы:**
- Job Stories
- Desired Outcomes
- Functional / Emotional Jobs

---

## 3.9 Comparison / Competitive framing sections

### Comparison Section
Сравнение с альтернативами.

**Задача:** помочь пользователю выбрать вас.  
**Когда использовать:** если рынок насыщен альтернативами.

**Типы:**
- Vs Competitors
- Vs In-House
- Vs Manual Process
- Vs Status Quo
- Alternative Comparison Grid

---

### Comparison Table
Табличная форма сравнения.

**Задача:** дать структурное сравнение по критериям.  
**Когда использовать:** для тарифов, конкурентов, feature parity.

**Типы:**
- Feature Comparison Table
- Plan Comparison Table
- Competitor Table
- Old Way vs New Way

---

### Why Switch Section
Почему стоит перейти сейчас.

**Задача:** мотивировать смену текущего подхода.  
**Когда использовать:** когда пользователь уже использует альтернативу.

**Типы:**
- Migration Benefit Block
- Switching Incentive
- Replace Legacy Workflow

---

Секция с ценой или тарифами.

- ROI Calculator

### Guarantee Section
Гарантии.

**Задача:** уменьшить страх покупки.  
**Когда использовать:** если есть сильное сопротивление или высокий чек.

**Типы:**
- Money-Back Guarantee
- Satisfaction Guarantee
- Performance Guarantee
- Deadline Guarantee

## 3.12 Objection Handling sections

Часто задаваемые вопросы.

**Задача:** снять типовые возражения и дать детали.  
**Когда использовать:** почти всегда.

**Типы:**
- Accordion FAQ
- Category FAQ
- Pricing FAQ
- Technical FAQ

---

Отдельная секция возражений.

**Задача:** адресно снять ключевые страхи.  
**Когда использовать:** для сложных и дорогих офферов.

# 7. Как выбирать секции осознанно

При проектировании лендинга каждую секцию нужно проверять вопросом:

## 1. Какую задачу она решает?
Если секция не решает конкретную задачу, она не нужна.

## 2. На каком этапе воронки она работает?
- Захват внимания
- Объяснение проблемы
- Раскрытие решения
- Доказательство
- Снятие возражений
- Конверсия

## 3. Что произойдет, если убрать эту секцию?
Если ничего критичного — вероятно, секция лишняя.

## 4. Это новая информация или повтор?
Лендинг должен продвигать пользователя вперед, а не повторять одно и то же.

## 5. Это про продукт или про пользователя?
Хороший лендинг объясняет продукт через выгоду для пользователя.

# 8. Частые ошибки при сборке лендинга

## 1. Слишком много секций
Избыточное количество блоков размывает фокус и ухудшает конверсию.

## 2. Повтор одной и той же мысли
Если 4 секции подряд говорят одно и то же разными словами — структура слабая.

## 3. Слишком ранний pricing
Иногда цену показывают до того, как раскрыта ценность.

## 4. Мало social proof
Даже хороший оффер проигрывает без подтверждения доверия.

## 5. Нет связки Problem → Solution
Пользователь должен видеть логический переход от боли к решению.

## 6. Features без Benefits
Функции сами по себе редко продают.

## 7. CTA без прогрева
Если пользователь не понял ценность, CTA не сработает.

## 8. Нет фильтрации аудитории
Когда страница пытается говорить со всеми сразу, она теряет силу.

# 10. Быстрый чек-лист перед запуском лендинга

## Структура
- Есть понятный Hero
- Сразу понятен оффер
- Есть логическая последовательность блоков
- Секции не дублируют друг друга
- Есть финальный CTA

## Маркетинг
- Проблема обозначена ясно
- Решение описано понятно
- Выгоды сформулированы через результат
- Есть доказательства доверия
- Есть обработка возражений

## UX
- CTA заметны
- Лейаут не перегружен
- Визуальная иерархия сильная
- Скролл логичен
- Мобильная версия не ломает смысл

## Коммерция
- Ясно, что получает клиент
- Ясно, что делать дальше
- Цена или путь к получению цены понятен
- Есть снижение риска
- Есть форма, заявка или контактный CTA

---

# 11. Итог

Хороший лендинг — это не набор красивых блоков, а логически выстроенная система убеждения.  
Каждая секция должна работать на конкретную задачу: внимание, понимание, доверие, доказательство или конверсию.

Сильный лендинг не перегружает.  
Он последовательно отвечает на вопросы пользователя:

1. Что это?  
2. Для кого это?  
3. Почему это мне нужно?  
4. Почему вам можно верить?  
5. Почему именно сейчас?  
6. Что мне делать дальше?  

Если страница отвечает на эти вопросы ясно и последовательно, структура собрана правильно.
