# AGENTS.md — контекст проекта WebCode и правила для агентов

Этот файл является стартовой точкой для всех AI-агентов, работающих над проектом.
Он задаёт приоритет документов, обязательные правила, рабочий workflow и минимальный стандарт качества.

## 1. О проекте

**WebCode** — коммерческий лендинг, продающий услугу запуска сайта и CRM под ключ для сервисного и экспертного бизнеса в Узбекистане.

**Ключевое предложение:**
Сайт + CRM + Telegram/WhatsApp + Домен/Хостинг + Поддержка в одном понятном запуске.

## 2. Технологический стек

Подробности в `Guides/tech_stack.md`.

- **Core:** Next.js 16 (App Router) + TypeScript
- **Styling:** Tailwind CSS v4 + shadcn/ui (New York style)
- **i18n:** next-intl v4.4 (`RU / UZ / EN`)
- **Forms:** react-hook-form + zod
- **Package Manager:** npm

## 5. Критические правила

### Всегда запрашивать разрешение перед

- изменением конфигурационных файлов: `package.json`, `tsconfig.json`, `next.config.*`, `eslint.config.*`
- установкой новых зависимостей или крупных наборов компонентов
- удалением файлов или папок
- крупными архитектурными изменениями, которые меняют структуру проекта

### Можно делать без дополнительного согласования в рамках задачи

- создавать обычные проектные файлы компонентов, секций, утилит и документов
- редактировать существующие файлы реализации
- добавлять локализации, контентные файлы и вспомогательные UI-компоненты
- улучшать структуру кода, если это не требует смены архитектуры или зависимостей

### Категорически запрещено

- начинать работу без изучения документации
- нарушать стандарты качества: `0` ошибок TypeScript и ESLint
- настраивать компоненты в местах использования, если это разрушает переиспользуемость
- устанавливать UI-компоненты без проверки через `shadcn MCP`
- забывать обновлять документацию после значимых задач
- хардкодить пользовательские тексты прямо в UI-компонентах
- использовать фейковые отзывы, кейсы или неподтверждённые метрики
- ломать мультиязычность ради локального удобства в одной секции
- самостоятельно добавлять новый контент, поясняющие тексты, смысловые блоки или продуктовые формулировки без явного запроса пользователя
- использовать любые документы, кроме `WebCode_landind_PRD.md`, как основной источник контента, текстов и продуктовых формулировок
- использовать `webcode_content_only.md` как актуальную базу контента
- считать задачу завершённой или закрывать её без явного подтверждения пользователя

## 6. Языковые правила

- Всегда отвечать пользователю на русском языке.
- Не переводить технические термины, команды и пути к файлам.
- Объяснения, комментарии и UI-тексты переводить на язык локали.

### Правило для `UZ`

- Все новые тексты для `UZ` создавать как полноценную локализацию для пользователей в Узбекистане, а не как буквальный перевод с русского.
- При создании и редактировании `UZ` текстов учитывать нормы современного узбекского языка, естественный порядок слов и коммерческий стиль, принятый в Узбекистане.
- Избегать русизмов, кальки, смешения с русским и неестественных формулировок.
- Если русская формулировка плохо переносится на узбекский, адаптировать смысл естественно для узбекского языка, сохраняя продуктовый смысл, а не структуру оригинала.

## 7. Правила проектирования и реализации

### Обязательные guide-документы по типу задачи

- Для структуры лендинга, состава секций, sequence блоков и layout-паттернов сначала сверяться с `Guides/guide_landing_layouts_PRO.md`.
- Для visual execution, typography, spacing, cards и UI quality сверяться с `Guides/landing_ui_standards.md`.
- Для responsive и mobile-задач сверяться с `Guides/mobile_and_responsive.md`.
- Для текстов и локализаций сверяться с `Guides/localization_standards.md`.
- Для CTA, формы заявки, messenger links и аналитических точек сверяться с `Guides/conversion_standards.md`.
- Для задач по подключению формы к `Notion` и проверке записи лидов сверяться с `Guides/notion_leads_integration.md`.
- Для задач по уведомлениям о новых заявках в `Telegram` через `bot + group` сверяться с `Guides/telegram_bot_group_setup.md` как со стандартным guide для повторного подключения на новом сайте.
- Для финальной UI-проверки использовать `Guides/accessibility_and_quality.md` вместе с `Guides/done_criteria.md`.

### i18n

- Проектировать все секции сразу под `RU / UZ / EN`.
- Не размещать UI-тексты напрямую в JSX, если для них предполагается локализация.
- Учитывать, что длина строк в разных языках отличается.
- Не делать layout, который держится только на одной длине заголовков или одной локали.

### Компоненты

- Каждая крупная секция лендинга должна быть отдельным компонентом.
- Повторяемые паттерны (`Section`, `CTA`, карточки, `badge`, `section heading`) выносить в переиспользуемые компоненты.
- Не смешивать в одном компоненте layout, контент и интеграционную логику без необходимости.
- Контент должен быть максимально отделён от представления.

### Контентные заглушки

- Если реальных кейсов, отзывов, цифр или логотипов клиентов нет, использовать только честные временные заглушки.
- Все временные заглушки должны быть явно распознаваемы как временные.
- Нельзя придумывать достижения, клиентов или результаты ради заполнения секции.
- Кейсы, тарифы, CTA, FAQ и оффер сверять с PRD, а не с устаревшими документами.

### SEO и аналитика

- Каждый экран и каждая локаль должны иметь корректные `metadata`.
- Готовить `title`, `description`, `Open Graph` и базовую semantic structure.
- CTA и формы должны быть готовы к аналитической разметке событий.
- Локализованные страницы должны учитывать `canonical` и `alternate`.
- Для `RU / UZ / EN` обязателен корректный `html lang` на уровне server-rendered документа, а не только в metadata.
- Для App Router не использовать чтение request headers как основной способ выставить `html lang` для локализованных страниц, если это переводит лендинг в unnecessary dynamic rendering.
- Для новых мультиязычных лендингов на Next.js локали должны изначально проектироваться через отдельные root layouts или route groups, чтобы `html lang` оставался корректным и страницы могли собираться как `SSG`.

## 8. Рекомендуемый workflow

1. Изучить `AGENTS.md`.
2. Изучить `WebCode_landind_PRD.md` или нужный документ из `Docs/PRD/`.
3. Если задача связана со структурой лендинга, проверить `Guides/guide_landing_layouts_PRO.md`.
4. Проверить профильные guide-документы по типу задачи.
5. Проверить `Roadmap.md`, чтобы понять текущий этап.
6. Выполнить задачу в логичном порядке:
   - `Discovery`
   - `Structure`
   - `Visual direction`
   - `App scaffold`
   - `Sections`
   - `Forms / CRM hooks`
   - `i18n`
   - `SEO`
   - `QA`
7. После изменений обновить `Roadmap.md`, если это требуется.
8. Зафиксировать важные изменения в `log.md`.

## 9. Обязательные инструменты

1. **Context7 MCP** — для актуальной документации библиотек.
2. **shadcn MCP** — для поиска и проверки UI-компонентов.
3. **stitch MCP** — для переноса дизайна страниц в проект из Google Stitch.

## 10. Минимальный чеклист проверки

- `npm run lint` — прогнать ESLint.
- `npm run build` — убедиться, что проект собирается и нет ошибок типов/конфигурации.
- `npm run dev` — локально проверить рендер страницы, шрифты и `metadata`, когда это уместно.

Дополнительно обязательно:
- проверить `mobile viewport`
- проверить все основные `CTA`
- проверить работу формы заявки
- проверить `RU / UZ / EN`
- проверить `logo`, `favicon` и `metadata`
- проверить `html lang` для каждой локали в итоговом HTML-документе
