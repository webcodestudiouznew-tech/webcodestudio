# AGENTS.md — Контекст проекта WebCode и правила для агентов.

Этот файл содержит ключевую информацию для всех AI-агентов, работающих над проектом. Используйте его для понимания целей, стека и правил.

## 1. Описание Проекта
**WebCode** — это коммерческий лендинг, продающий услугу запуска сайта и CRM под ключ для сервисного и экспертного бизнеса в Узбекистане.
**Ключевое предложение:** Сайт + CRM + Telegram/WhatsApp + Домен/Хостинг + Поддержка в одном понятном запуске.

## 2. Технологический Стек
Подробности в `Guides/tech_stack.md`
- **Core:** Next.js 16 (App Router) + TypeScript
- **Styling:** Tailwind CSS v4 + shadcn/ui (New York style)
- **i18n:** next-intl v4.4 (Поддержка RU, UZ, EN)
- **Forms:** react-hook-form + zod
- **Package Manager:** npm

## 6. Правила проектирования и реализации

### i18n
- Проектировать все секции сразу под `RU / UZ / EN`.
- Не размещать UI-тексты напрямую в JSX, если для них предполагается локализация.
- Учитывать, что длина строк в разных языках отличается.
- Не делать layout, который держится только на одной длине заголовков.

### Компоненты
- Каждая крупная секция лендинга должна быть отдельным компонентом.
- Повторяемые паттерны (`Section`, `CTA`, карточки, badge, section heading) выносить в переиспользуемые компоненты.
- Не смешивать в одном компоненте layout, контент и интеграционную логику без необходимости.
- Контент должен быть максимально отделён от представления.

### Контентные заглушки
- Если реальных кейсов, отзывов, цифр или логотипов клиентов нет, использовать только честные временные заглушки.
- Все временные заглушки должны быть явно распознаваемы как временные.
- Нельзя придумывать достижения, клиентов или результаты ради заполнения секции.
- Кейсы, тарифы, CTA, FAQ и оффер сверять с PRD, а не с устаревшими документами.

### SEO и аналитика
- Каждый экран и локаль должны иметь корректные metadata.
- Нужно готовить `title`, `description`, Open Graph и базовую semantic structure.
- CTA должны быть готовы к аналитической разметке событий.
- Локализованные страницы должны учитывать `canonical` и `alternate` связи.

## 7. Рекомендуемый workflow

Порядок работы по умолчанию:

1. Изучить `AGENTS.md`, затем `WebCode_landind_PRD.md` или нужный документ из `Docs/PRD/`.
2. Проверить `Roadmap.md`, чтобы понять текущий этап.
3. Выполнить задачу в логичном порядке:
   - Discovery
   - Structure
   - Visual direction
   - App scaffold
   - Sections
   - Forms / CRM hooks
   - i18n
   - SEO
   - QA
4. После изменений обновить `Roadmap.md` при необходимости.
5. Зафиксировать важные изменения в `log.md`.

## 9. Минимальный чеклист для проверки:

npm run lint — прогнать ESLint.
npm run build — убедиться, что проект собирается (поймает ошибки типов/конфигурации).
npm run dev — локально посмотреть, что страница рендерится и шрифты/metadata работают (ручная проверка в браузере).

Дополнительно обязательно:
- Проверить mobile viewport.
- Проверить все основные CTA.
- Проверить работу формы заявки.
- Проверить RU / UZ / EN.
- Проверить logo, favicon и metadata.
